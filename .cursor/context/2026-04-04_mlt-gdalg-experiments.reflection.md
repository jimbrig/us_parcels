# Reflection: MLT Tiles via tileserver-rs + GDALG Pipeline Experiments

**Date**: 2026-04-04
**Scope**: Two geospatial experiments (MLT vector tiles, GDALG declarative pipelines) plus compose stack stabilization

## Direction Changes

### tileserver-rs over Martin for MLT
Martin v1.3.1's "MLT decoder" is for reading pre-built MLT tiles, not on-the-fly MVT-to-MLT transcoding. tileserver-rs (`ghcr.io/vinayakkulkarni/tileserver-rs`) transcodes MVT PMTiles to MLT on request via the `.mlt` file extension — confirmed working with Atlanta parcels: 411KB MVT -> 260KB MLT (37% smaller). Moving forward, tileserver-rs is the MLT tile server, not Martin.

### Planetiler is OSM-specific, not general-purpose
Planetiler's `generate-custom` YAML schema supports `osm`, `shapefile`, `geopackage`, `geojson` source types — not GeoParquet or FlatGeobuf directly. Its parquet reader is Overture-specific. Additionally, Planetiler v0.10.2 requires Java 21 (system only has Java 11). Abandoned this path entirely in favor of tileserver-rs for MLT.

### GDALG pipeline syntax
GDALG JSON files must be generated by `gdal vector pipeline` CLI — not hand-crafted. The write step is automatically stripped by GDAL when outputting to `.gdalg.json`. Hand-written `write` steps with dummy paths or `--format stream` cause cascading errors. Correct workflow: run the full pipeline command with the output path as `.gdalg.json` and GDAL generates the correct JSON.

## Corrections

- **Source layer names**: PMTiles layer names don't match what was assumed. `atlanta_downtown.pmtiles` has layer `lr_parcel_us` (from the GPKG source layer), not `atlanta_downtown`. `showcase/parcels.pmtiles` has layer `showcase`, not `parcels`. Always verify with `ogrinfo -so file.pmtiles` before referencing in map styles.
- **Martin crash loop**: Config referenced nonexistent table `parcel_raw`, tables without spatial indexes (`tiger.county`, `tiger.tract`), a PMTiles file that didn't exist yet, and an unrecognized `webui` key (removed in Martin v1.3.1). Martin treats unresolvable sources as fatal errors.
- **Maputnik proxy**: nginx proxy at `/maputnik/` can't serve SPAs with absolute asset paths. Must use direct port `:8888`.
- **pyarrow conflict**: `pyarrow >=19.0,<20` in pixi conflicts with GDAL 3.12.3's libarrow 20 dependency. Removed `pyarrow` from pixi deps — `libgdal-arrow-parquet` provides the GDAL-level Arrow/Parquet support without the Python pyarrow package.
- **Port confusion**: Tiles served through Martin only work via nginx proxy on `:8080`. The bun static server on `:8081` has no proxy for `/tiles/`. Map layers that reference `${base}/tiles/...` fail on `:8081`.

## Learned Preferences

- **Don't inline layer styles in HTML JS** — use MapLibre style JSON files (`config/styles/parcels.json`). Styles are tightly coupled to tile data and should be managed alongside the tile service.
- **Frontend architecture**: Single HTML files with CDN script tags and inline CSS/JS is not the correct structure. Dependencies should be managed via package.json/bun, bundled properly. This needs restructuring.
- **Use `ogrinfo -so` and `pixi run ogr2ogr` for CLI validation** before writing GDALG files or pipeline scripts. Test the direct VSI path first, confirm field names and driver, then generate the GDALG JSON.
- **Docker for Java**: Use `eclipse-temurin:21-jre` image rather than installing Java locally. Pixi should not include Java.
- **tileserver-rs config**: Needs explicit `[[sources]]` in TOML config; auto-detect from `/data/` directory alone does not work. Config must specify `path`, `type`, `id` per source.

```
  geoparquet/tiger/state=13/county=121/edges.parquet  (GDALG output, 0.55MB)
  sample/showcase/parcels.pmtiles  (7KB, layer: showcase, z8-16, ~3 parcels only)
```

### GPKG characteristics
- `LR_PARCEL_NATIONWIDE_FILE_US_2026_Q1.gpkg`: 93.7GB, layer `lr_parcel_us`
- Full state extract via bbox filter takes 30-60min (no spatial index on the GPKG)
- Record warnings (e.g., `saledate` field) are benign data quality issues

### tileserver-rs API
- TileJSON: `GET /data/{source}.json`
- MVT tile: `GET /data/{source}/{z}/{x}/{y}.pbf`
- MLT tile: `GET /data/{source}/{z}/{x}/{y}.mlt`
- Health: `GET /health`
- Catalog: `GET /data.json`
- Content types: `application/x-protobuf` (MVT), `application/vnd.maplibre-vector-tile` (MLT)
